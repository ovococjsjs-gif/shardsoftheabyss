# Fiction Craft — Универсальная система работы с прозой

Синтез шести профессиональных систем: story-skills, Claude-Book, creative-writing-skills,
fiction (howells), The Crucible, Claude-Code-Novel-Writer + ai-fiction-writer.

---

## §0. ДИАГНОСТИКА: определи режим до любого действия

| Запрос пользователя | Режим | Переходи к |
|---|---|---|
| «Напиши сцену / главу» | ГЕНЕРАЦИЯ | §5 → §7 |
| «Улучши / перепиши мой текст» | РЕДАКТУРА | §1 → §6 → §7 |
| «Проверь на ИИ-маркеры» | АУДИТ | §1 + §7 |
| «Создай персонажа / мир» | ПРЕПРОДАКШН | §2 |
| «Проверь логику / непрерывность» | КОНТРОЛЬ | §4 |
| «Обсудим структуру / сюжет» | АРХИТЕКТУРА | §2.1 + §3 + §8 |
| «Работаем над романом / новеллой» | ПОЛНЫЙ ЦИКЛ | §2 → §3 → §4 → §5 → §6 → §7 |

## §1. СТИЛОМЕТРИЧЕСКИЙ РАДАР: красные флаги ИИ-прозы

ИИ-текст детектируется с точностью 89% по этим признакам. Запусти проверку
**до** любой генерации и **после** любого черновика.

### Красные флаги (удалять хирургически)
- **Синтаксическая монотонность** — предложения одной длины
- **On-the-nose диалог** — персонажи говорят именно то, что думают
- **RLHF-позитивизм** — искусственная бесконфликтность, «добрые выводы»
- **Лексическая вычурность** — сложные слова там, где человек сказал бы проще
- **ИИ-штампы**: «глубокий вздох», «уголки губ», «тихая комната», «он почувствовал, что...», «её сердце сжалось»
- **Абстрактные прилагательные**: «красивый», «страшный», «странный» без физической конкретики
- **Низкая перплексивность** — предсказуемые продолжения, нет ритмических сбоев
- **Морализаторство в финале** — сцена «подводит итог» вместо того чтобы оставлять открытый вопрос

### Параметры человеческого текста (цель)
- Высокая амплитуда длин предложений (короткие и длинные вперемешку)
- Амбивалентные, тёмные, противоречивые эмоции
- Сенсорная конкретность: запах, фактура, температура — не эпитеты
- Субтекст: персонажи говорят одно, думают другое
- Нарративные сбои, необъяснённые детали, открытые вопросы в конце сцены

## §2. БИБЛИЯ ПРОЕКТА (ПОСТОЯННАЯ — не меняется в процессе работы)

Библия — это **контракт**. Нарушение однажды заявленных правил разрушает
доверие читателя быстрее любой стилистической слабости.

### 2.1 Руководство по стилю (Writing Style Guide)

Зафиксировать и хранить:
```yaml
жанр: [ЖАНР]
тональность: [ТОНАЛЬНОСТЬ]
литературные_влияния: [АВТОРЫ/КНИГИ]
синтаксический_ритм: [ОПИСАНИЕ]
особенности_диалога: [ОПИСАНИЕ]
POV: [первое лицо / третье ограниченное / всеведущее]
темп: [быстрый / медленный / переменный]
запрещённые_слова: [СПИСОК 20+ клише и штампов проекта]
целевая_аудитория: [ОПИСАНИЕ]
```

### 2.2 Профили персонажей

Для каждого персонажа — обязательные поля:

| Поле | Описание |
|---|---|
| Базовый страх | Не «боится пауков» — глубинная экзистенциальная уязвимость |
| Внутренний конфликт | Противоречие между желанием и необходимостью |
| Речевые маркеры | Словарь, ритм, что НИКОГДА не говорит |
| Поведение под давлением | Как ломается — не как держится |
| Психотип | Enneagram или Big Five (для предотвращения размывания) |
| Статус жизни | Живой / мёртв / неизвестно (для движка непрерывности) |
| Знания персонажа | Что он знает к началу каждой главы |

### 2.3 Кодекс мира (World Codex)

Хронология, география, законы физики/магии, социальные системы, технологии.
Каждый заявленный закон — твёрдое правило. Экстраполируй следствия, не изобретай
новые правила по мере надобности (3-й закон Сандерсона).

> Подробные шаблоны файлов: `references/continuity-engine.md` → раздел «Шаблоны»

## §3. СОСТОЯНИЕ ГЛАВЫ (ИЗМЕНЯЕМОЕ — версионируется после каждой главы)

После каждой главы фиксировать:

```yaml
глава: N
дата_в_мире: [внутримировая дата]
место: [локация]
персонажи_в_сцене: [список]
изменения_состояний:
  - персонаж: [имя]
    изменение: [что изменилось — знания, физическое состояние, отношения]
объекты_введённые: [предметы/факты, введённые впервые]
открытые_вопросы: [что заявлено, но не закрыто]
ружья_введённые: [что будет использовано позже]
```

Архив состояний хранится — можно откатиться к любой точке.

## §4. ДВИЖОК НЕПРЕРЫВНОСТИ (детерминированные проверки)

Запускать перед финальной редактурой каждой главы. Обрабатывает противоречия
как компилятор — остановить и исправить, не «пройти мимо».

### Четыре обязательные проверки

**Проверка 1 — Смерти**
Персонаж помечен как мёртвый? Не появляется в последующих сценах без
сверхъестественного объяснения, явно встроенного в логику мира.

**Проверка 2 — Обещания/Выплаты (Promises & Payoffs)**
- Payoff не может приходить раньше своего Setup
- Каждое введённое «ружьё» — либо стреляет, либо сознательно снято с гвоздя
- Масштаб события ∝ глубина предзнаменований

**Проверка 3 — Открытые вопросы**
Каждый вопрос, заявленный как значимый — закрыт к концу истории.
Незакрытые вопросы маркированы как намеренные (серия, открытый финал).

**Проверка 4 — Состояние объектов и знаний**
Персонаж не может знать то, что ещё не произошло в его временной линии.
Уникальный объект не может быть одновременно в двух местах.

> Полные таблицы отслеживания и шаблоны: `references/continuity-engine.md`

## §5. ПОСЛОЙНАЯ ГЕНЕРАЦИЯ СЦЕНЫ (5 этапов)

**Золотое правило**: никогда не генерировать сцену целиком за один запрос.
Всегда — последовательно, слой за слоем.

### ЭТАП 1 — Сенсорное заземление
```yaml
Роль: Мастер сенсорного описания
Задача: Написать физическое пространство сцены
Ограничения:
  - Только конкретные сенсорные маркеры: звуки, запахи, фактуры, температура, свет
  - ЗАПРЕЩЕНО: внутренние монологи, диалоги, предыстория, продвижение сюжета
  - ЗАПРЕЩЕНО: абстрактные прилагательные ("красивый", "страшный")
  - Заменять абстракции физическими метафорами и действиями
```

### ЭТАП 2 — Проекция сознания
```yaml
Роль: Психологический романист
Задача: Интегрировать поток мыслей персонажа в физическое пространство (Этап 1)
Ограничения:
  - Отражать тревогу, скрытые мотивы, интеллектуальный уровень персонажа
  - ЗАПРЕЩЕНО: диалоги, активные действия
  - Фокус: интерпретация увиденного через призму его страхов и желаний из Библии
```

### ЭТАП 3 — Динамический диалог
```yaml
Роль: Сценарист и диалоговый коуч
Задача: Развернуть диалог как битву подтекстов, не обмен информацией
Ограничения:
  - Диалог РАЗВИВАЕТ конфликт, не разрешает его
  - ЗАПРЕЩЕНО: on-the-nose реплики
  - ОБЯЗАТЕЛЬНО: разные скрытые цели у каждого персонажа
  - Внедрить: перебивания, недосказанности, паузы, речевые тики
  - ЗАПРЕЩЕНО: вежливые банальности, info-dump в диалоге
```

### ЭТАП 4 — Финальный синтез и темпоритм
```yaml
Роль: Шеф-редактор
Задача: Объединить слои 1–3 в единую сцену
Алгоритм:
  - В моменты напряжения: укорачивать предложения, убирать ремарки
  - В моменты рефлексии: расширять синтаксические конструкции
  - Устранить лексические повторы
  - Проверить физическую логику перемещения персонажей в пространстве
  - Мини-клиффхэнгер в финале: неразрешённое противоречие, крючок
```

### ЭТАП 5 — АнтиИИ-очистка (ОБЯЗАТЕЛЬНЫЙ, всегда последний)

Запускать на каждом финальном тексте без исключений.

```yaml
Роль: Стилистический хирург + Перплексивный гейт
Чек-лист:
  1. ИИ-штампы: удалить полностью, не заменять другими штампами
  2. Абстрактные прилагательные → конкретные действия/детали
  3. Физически невозможные движения (улыбнулась и одновременно вздохнула)
  4. Морализаторство или «подведение итогов» в финале сцены
  5. RLHF-позитивизм: искусственное смягчение конфликта
  6. On-the-nose диалог, если просочился
  7. Синтаксическая монотонность: выровнять амплитуду длин
  8. ПЕРПЛЕКСИВНЫЙ ГЕЙТ: найти предложения с предсказуемым продолжением
     и переписать с неожиданным, но органичным поворотом
```

> **Метод Франкенштейна**: генерировать 3 версии одной сцены → компилировать лучшие
> фрагменты → накладывать авторский стиль. Неудачные версии — негативное пространство.

## §6. МНОГОРОЛЕВАЯ РЕДАКТУРА

При работе с текстом пользователя применять все четыре роли:

| Роль | Фокус | Вопросы |
|---|---|---|
| **Читатель** | Эмоциональный отклик | Скучно? Путаница? Крючок работает? |
| **Автор** | Интенция и голос | Это то, что ты хотел сказать? Голос твой? |
| **Редактор** | Структура и темп | Арки закрыты? Ритм правильный? Непрерывность? |
| **Стилист** | Проза и перплексивность | Антиштамп. АнтиИИ. Ритм предложений. |

**Порядок отчёта**: сначала назвать конкретные находки по §1 (радар),
потом предложить конкретные правки с объяснением через какую роль смотришь.

## §7. АНТИИИ + ПЕРПЛЕКСИВНЫЙ ГЕЙТ (детально)

### Перплексивный гейт (из Claude-Book)
Найти предложения с низкой перплексивностью — те, где любой читатель
знает, что будет дальше:

❌ «Она открыла дверь и вошла в тёмную комнату.»
✅ «Дверь не заскрипела — она открылась с таким спокойствием, будто её открывали
   каждую ночь. Ждали.»

❌ «Он понял, что влюблён.»
✅ «Через три часа после разговора он всё ещё знал, где именно она стояла.»

Алгоритм гейта:
1. Прочитать предложение
2. Спросить: «Что будет в следующих словах — понятно до их прочтения?»
3. Если да — переписать с сохранением смысла, но изменением направления

### Запрещённые конструкции (универсальный список)
```
глубокий вздох / тяжело вздохнул
уголки губ / краешки губ / тонкая улыбка
комната была тихой / повисла тишина
он/она почувствовал(а) что... / осознал что...
её/его сердце сжалось / ёкнуло / подпрыгнуло
что-то в нём/ней изменилось
слёзы навернулись на глаза
он не знал что сказать
несмотря на всё / тем не менее / однако
[имя] кивнул(а) / покачал(а) головой (как ремарка)
```

## §8. НАРРАТИВНЫЕ ТЕХНИКИ (краткий справочник)

Подробнее: `references/narrative-techniques.md`

### Темпоритм
- Экшен: ультракороткие предложения, дробные абзацы
- Рефлексия: многосоставные конструкции
- «Тикающие часы»: жёсткий временной лимит = максимальный драматизм
- Breather scenes: сцены-передышки между пиками — обязательны

### Show, Don't Tell
- ❌ «Джек испытывал страх»
- ✅ «Когда шаги зазвучали ближе, спазм сжал желудок. Джек прижался к стене —
  щека почувствовала холодный крошащийся кирпич.»

### Субтекст
- Стихомития (rapid fire) = скрытая агрессия
- Loaded questions = вопросы с двойным дном
- Pope in the Pool: сложная экспозиция через захватывающую сцену

### Три закона Сандерсона (миростроительство)
1. Магия решает конфликт только если читатель знает правила
2. Слабости и цена > сама сила
3. Углублять следствия заявленного, не изобретать новое

## §9. БЫСТРЫЙ СТАРТ ПО РЕЖИМАМ

### Пользователь даёт текст на редактуру
1. Запустить §1 (радар) — назвать конкретные найденные маркеры
2. Применить §6 (четыре роли) — структурированный отчёт
3. Применить §7 (перплексивный гейт) к проблемным фрагментам
4. При необходимости — пересобрать через §5

### Пользователь просит написать сцену
1. Запросить: локацию, персонажей (из Библии §2), цель сцены, место в сюжете
2. Применить §5, Этапы 1→2→3→4
3. Обязательно §5 Этап 5 (антиИИ) + §7 (гейт)

### Пользователь начинает роман/новеллу с нуля
1. Собрать Библию §2 (стиль + персонажи + кодекс мира)
2. Создать реестр: обещания, ружья, открытые вопросы
3. После каждой главы: заполнять §3 (состояние)
4. Периодически: §4 (движок непрерывности)

### Пользователь проверяет рукопись на качество
1. §1 (радар) по всему тексту
2. §4 (движок непрерывности) — все четыре проверки
3. §6 (все четыре роли) — структурированный разбор
4. Приоритизировать правки по уровню: структурные > сцены > проза > стиль
